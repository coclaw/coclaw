# v0.2 实时通信架构整改方案

> 日期：2026-03-11
> 状态：已确认
> 涉及模块：ui, server, plugins/openclaw

## 一、背景与问题

v0.1（MVP）的实时通信链路存在以下结构性问题：

1. **UI 侧 WebSocket 为一次性连接**：每次操作（发消息、加载 session 列表、reconcile）都创建新 WS 连接，需要先 REST 获取 ticket 再握手，延迟高、开销大、无法接收服务器推送。
2. **通信逻辑与 UI 组件耦合**：ChatPage.vue 同时承担连接管理、RPC 调度、消息格式化、流式响应处理、session 轮换检测等职责，难以复用和测试。
3. **Plugin→Server WS 无心跳**：中间网络设备（NAT/LB）可能因空闲超时断连，Plugin 侧无法及时感知"假死连接"。
4. **缺乏响应式状态推送**：Bot 状态变更、其他 session 的新消息等无法实时推送到 UI，用户需主动刷新。

## 二、整改目标

1. UI 与 Server 之间建立 **per-bot 持久 WS 长连接**（自动重连、心跳保活）
2. 通信层从 UI 组件中彻底剥离，建立 **Service → Store → Component** 三层架构
3. WS 认证从一次性 ticket 改为 **session cookie**
4. Plugin→Server 增加**应用层心跳**
5. Store 层设计为**事件驱动的响应式更新**，为后续状态推送做好架构准备

### 不在本轮范围

- base64 附件传输优化（后续单独处理）
- Server 侧安全防护（payload 限制、速率控制）
- Channel 消息管线适配（非图片附件支持）
- UI 页面布局与交互方式变更
- SSE 通道移除（保留，P2P 场景下仍可能需要平台级通知通道）

## 三、架构设计

### 3.1 分层总览

```
┌─────────────────────────────────────────────────────────┐
│                    Vue 组件层                             │
│  ChatPage / MainList / ManageBotsPage / ...              │
│  职责：渲染 + 用户交互                                     │
│  原则：只从 Store 读数据，通过 Store actions 触发操作        │
└────────────────────────────┬────────────────────────────┘
                             │ reactive (Pinia)
┌────────────────────────────┴────────────────────────────┐
│                    Pinia Store 层                         │
│  botsStore / sessionsStore / chatStore                    │
│  职责：状态管理、响应事件更新数据、暴露 actions             │
│  原则：Single Source of Truth                            │
└────────────────────────────┬────────────────────────────┘
                             │ 调用方法 / 监听事件
┌────────────────────────────┴────────────────────────────┐
│                    Service 层                             │
│  BotConnectionManager → BotConnection (per bot)          │
│  职责：WS 生命周期、RPC 协议、心跳、重连、事件分发          │
│  原则：无 Vue 依赖，纯 JS，可独立测试                      │
└────────────────────────────┬────────────────────────────┘
                             │ WebSocket (per bot)
                     CoClaw Server
```

### 3.2 Service 层设计

#### BotConnection（per-bot 连接实例）

每个已绑定的 Bot 对应一个 `BotConnection` 实例，负责：

- **连接生命周期**：建立 WS、自动重连（指数退避）、主动关闭
- **认证**：WS upgrade 时自动携带 session cookie，URL query 中携带 `botId`
- **心跳保活**：应用层 ping/pong（与 v0.1 的 UI 侧心跳机制一致）
- **RPC 协议**：request/response（含两阶段 agent RPC）、pending 管理、超时处理
- **事件分发**：接收 server 推送的 event 帧，通过 EventEmitter 向上层分发
- **状态暴露**：`connecting` / `connected` / `disconnected` 等连接状态

关键接口（概念性）：

```js
class BotConnection extends EventEmitter {
  constructor(botId, options)

  // 生命周期
  connect()
  disconnect()
  get state()           // 'connecting' | 'connected' | 'disconnected'

  // RPC
  request(method, params, options)  // options: { onAccepted, onUnknownStatus, timeout }

  // 事件（继承 EventEmitter）
  // emit('agent', payload)         — agent 流式事件
  // emit('state', newState)        — 连接状态变更
  // emit('error', error)           — 连接错误
  // emit('event', { event, payload }) — 其他 server 推送事件
}
```

#### BotConnectionManager（全局单例）

管理所有 BotConnection 实例的生命周期：

```js
const manager = {
  // 为指定 bot 建立连接（幂等）
  connect(botId),

  // 断开指定 bot 连接
  disconnect(botId),

  // 为所有已绑定 bot 建立连接
  connectAll(botIds),

  // 断开所有连接
  disconnectAll(),

  // 获取指定 bot 的连接实例
  get(botId),           // → BotConnection | undefined

  // 获取所有连接的状态
  getStates(),          // → Map<botId, state>
}
```

**连接建立时机**：用户登录后、bots 列表加载完成时，调用 `connectAll()` 为所有已绑定 bot 建立连接。新绑定 bot 时追加 `connect(botId)`，解绑时 `disconnect(botId)`。

**与未来 P2P 的关系**：`BotConnection` 封装了传输层细节。未来 P2P 引入后，可以在 `BotConnection` 内部切换底层传输（server-relay WS vs P2P WS），上层 Store 和 Component 无感知。

### 3.3 Store 层设计

#### botsStore（已有，需增强）

现有职责保留（bot 列表管理），新增：

- 从 `BotConnectionManager` 同步连接状态到 bot 的 `connectionState` 字段
- 绑定/解绑时触发 `BotConnectionManager.connect/disconnect`

#### sessionsStore（已有，需重构）

当前逻辑：自行创建临时 WS 连接加载 session 列表。

整改后：

- 通过 `BotConnectionManager.get(botId).request(...)` 调用 RPC，不再自行管理 WS
- session 列表按 botId 分组缓存
- 提供 action：`loadSessions(botId)`, `refreshSessions(botId)`
- 预留：后续接收 WS 推送的 session 变更事件，增量更新列表

#### chatStore（新增，从 ChatPage 中剥离）

当前状态：消息列表、streaming 状态、agent 事件处理等逻辑全部在 ChatPage.vue 的 data/methods 中。

整改后，chatStore 负责：

- 当前活跃 session 的消息列表（`messages`）
- streaming 状态（`isStreaming`, `currentRunId`）
- 消息发送 action：`sendMessage(botId, text, files, sessionOptions)`
  - 内部通过 `BotConnectionManager.get(botId)` 获取连接、发送 RPC、处理两阶段响应
- agent 事件处理：注册/注销 `agent` 事件监听，处理各 stream 类型（assistant/tool/thinking/lifecycle）
- 发送后 reconcile（刷新 session 列表等）
- 按 sessionId 的消息缓存（未来支持切换 session 时无需重新加载）

### 3.4 Server 端改动

#### WS upgrade 支持 session cookie 认证

当前 `attachBotWsHub` 中 UI 客户端通过 ticket 认证。整改后：

- WS upgrade 时从 HTTP 请求头中解析 cookie，手动调用 Express session 中间件加载 session
- 从 session 中提取 `userId`，从 URL query `?botId=xxx` 获取 `botId`
- 验证该用户确实绑定了该 bot
- **移除** ticket 相关逻辑（`uiTickets` Map、`POST /api/v1/bots/ws-ticket`、`createBotWsTicket()`）

session 过期处理：

- Server 在 WS 连接期间定期（或在收到 RPC 请求时）校验 session 有效性
- session 过期后向 UI 发送 `{ type: 'session.expired' }` 事件，并关闭连接
- UI 侧收到后跳转登录页

#### Bot 侧 WS 心跳支持

当前 server 对 bot WS 连接无 ping/pong。整改后：

- 为 bot 侧连接启用与 UI 侧相同的 WS 层 ping/pong 定时器
- 识别并响应 bot 发来的应用层 `{ type: 'ping' }` 消息（当前已支持，但 plugin 侧没有发送）

### 3.5 Plugin 端改动

#### realtime-bridge 增加应用层心跳

在 `serverWs` 连接建立后：

- 每 25 秒发送 `{ type: 'ping' }` 到 server
- 45 秒未收到任何消息则判定连接死亡，关闭并触发重连
- 机制与 UI 侧 `gateway.ws.js` 的心跳逻辑保持一致

## 四、关键设计决策

### D1: Per-bot 独立 WS 连接（而非多路复用）

**决策**：每个已绑定 bot 建立独立的 WS 连接，不使用单连接多路复用。

**理由**：
- 与未来 P2P 架构天然对齐：每条连接就是一条 "UI↔Bot 通道"，底层传输可独立切换
- 故障隔离：一个 bot 的连接问题不影响其他 bot
- 实现简单：无需多路复用的消息分发、ID 映射和流量交织
- 连接数可控：普通用户绑定 bot 数量有限（个位数级别）

### D2: 认证方式改用 session cookie

**决策**：WS 连接复用 HTTP session cookie 认证，移除 ticket 机制。

**理由**：
- 持久连接场景下一次性 ticket 概念不匹配，每次重连都要额外 REST 调用
- 浏览器 WS 连接自动携带同域 cookie，无需额外操作
- 与 REST API 认证方式一致，减少维护两套认证逻辑的负担
- Capacitor WebView 中 cookie jar 与 fetch/XHR 共享，WS 连接同样有效

**注意事项**：
- WS upgrade 请求不经过 Express 中间件栈，需手动解析 cookie 和加载 session
- 长连接期间需处理 session 过期场景

### D3: 连接在登录后立即建立

**决策**：用户登录且 bots 列表加载完成后，立即为所有已绑定 bot 建立 WS 连接。

**理由**：
- Bot 在线状态实时可知，无需等到进入聊天页
- 可随时接收 server 推送的事件（为第 11 点响应式需求做准备）
- 进入聊天页时连接已就绪，减少用户等待

### D4: 保留 SSE 通道

**决策**：现有 SSE 通道（`bot-status-sse.js` + `use-bot-status-sse.js`）保留不变。

**理由**：
- SSE 是 UI 与 Server 之间的平台级通道，与 per-bot WS 连接正交
- P2P 场景下 bot WS 走直连，但平台级通知（如 bot 上线/下线）仍需要独立通道
- 待 P2P 架构明确后再决定是否合并

## 六、实施阶段

整改按 5 个阶段执行，每个阶段产出可独立运行、可验证的系统。阶段间可由不同 session 分别执行。

### Stage 1: Server 端适配

**目标**：Server 支持 session cookie WS 认证 + bot 侧心跳，同时保持 ticket 机制兼容（过渡期）。

**改动**：
1. `bot-ws-hub.js`：UI WS upgrade 路径增加 session cookie 认证（与 ticket 并存，优先 cookie）
2. `bot-ws-hub.js`：为 bot 侧连接增加 WS 层 ping/pong 定时器
3. 对应单测更新

**验证**：
- 旧 UI（ticket 方式）仍能正常工作
- 新方式（cookie + botId query）能通过认证建立连接
- bot 侧连接有 ping/pong 心跳

**依赖**：无

### Stage 2: UI Service 层

**目标**：建立 BotConnection + BotConnectionManager，实现持久 WS 连接基础设施。

**改动**：
1. 新建 `services/bot-connection.js`（BotConnection 类）
2. 新建 `services/bot-connection-manager.js`（BotConnectionManager 单例）
3. 完整单测覆盖

**验证**：
- 单测覆盖连接生命周期、RPC 协议（含两阶段）、心跳、重连、事件分发
- 可通过手动调用验证与 server 的实际连接

**依赖**：Stage 1（server 支持 cookie 认证）

### Stage 3: UI Store 层重构

**目标**：sessionsStore 和新建 chatStore 接入 BotConnectionManager，不再自行管理 WS。

**改动**：
1. 新建 `stores/chat.store.js`（从 ChatPage 提取消息/streaming/agent 逻辑）
2. 重构 `stores/sessions.store.js`（通过 BotConnectionManager 调用 RPC）
3. 增强 `stores/bots.store.js`（集成 BotConnectionManager 生命周期）
4. 完整单测覆盖

**验证**：
- sessionsStore 能通过持久连接加载 session 列表
- chatStore 能完成消息发送、streaming 接收、reconcile 全流程
- bots 列表变更时连接自动同步

**依赖**：Stage 2

### Stage 4: UI Component 层简化

**目标**：ChatPage 等组件剥离通信逻辑，只从 Store 读写数据。

**改动**：
1. ChatPage.vue：移除 `rpcClient`、`ensureRpcClient`、`sendViaAgent`、`onAgentEvent`、`__reconcileMessages`、`__detectRotation` 等，改为调用 chatStore actions 并绑定 store state
2. MainList / session 列表相关组件：改为读取 sessionsStore
3. 移除 `services/gateway.ws.js` 及其测试
4. 移除 `services/bots.api.js` 中的 `createBotWsTicket()`

**验证**：
- 完整用户流程回归：登录 → 查看 bot 列表 → 进入聊天 → 发消息 → 接收 streaming 响应 → 切换 session
- 确认 gateway.ws.js 的所有调用方已迁移

**依赖**：Stage 3

### Stage 5: Plugin 心跳 + Server 清理

**目标**：Plugin 增加心跳，Server 移除 ticket 遗留代码。

**改动**：
1. `realtime-bridge.js`：`serverWs` 连接上启动应用层心跳
2. `bot-ws-hub.js`：移除 ticket 认证路径和 `uiTickets` Map
3. `routes/bot.route.js`：移除 `POST /api/v1/bots/ws-ticket` 端点
4. 更新 `docs/architecture/overview.md` 中的 API 清单

**验证**：
- Plugin 心跳正常工作，空闲期连接不被中间设备断开
- Server 无 ticket 相关代码残留
- 全链路回归测试通过

**依赖**：Stage 4（确认 UI 侧不再使用 ticket）

## 七、跨阶段执行指引

本方案设计为跨多个 session 执行。每个 session 开始时：

1. 阅读本文档确认当前应执行的 stage
2. 阅读对应 workspace 的 `STATUS.md` 了解最新状态
3. 执行该 stage 的改动
4. 通过对应的验证清单
5. 更新 `STATUS.md` 记录完成状态

每个 stage 完成的判定标准：
- 代码改动完成
- `pnpm check` 通过
- `pnpm test` 通过
- `pnpm coverage` 通过（或记录例外原因）
- `STATUS.md` 已更新

## 八、风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| Session cookie 在 Capacitor WebView 中不可用 | Stage 1 保持 ticket 兼容，Stage 4 验证 Capacitor 环境后再移除 |
| 持久连接增加 server 内存占用 | 当前用户量小，暂不优化；后续按需增加连接上限和空闲回收 |
| 重构过程中中间状态破坏现有功能 | 每个 stage 保持系统可用；Stage 1 保持 ticket 兼容 |
| chatStore 逻辑复杂度高（从 ChatPage 提取） | Stage 3 专注 store 逻辑和测试，Stage 4 再改组件 |
